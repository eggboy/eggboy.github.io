2. Key Concepts

Weight Memory

The model's parameters (weights) must reside in GPU memory (VRAM). For FP16/BF16 precision, each parameter occupies 2 bytes. Quantized models use fewer bytes — for example, MXFP4 uses just 0.5 bytes per parameter for quantized layers.

KV Cache

During autoregressive generation, the Key and Value tensors from self-attention are cached to avoid recomputation. The KV cache grows linearly with sequence length and concurrent requests. It is often the dominant memory consumer for long-context workloads.

Prefill vs. Decode

LLM inference has two phases:

  • Prefill (prompt processing): All input tokens are processed in parallel. This phase is compute-bound — it depends on GPU TFLOPS.
  • Decode (token generation): Tokens are generated one at a time. This phase is memory-bandwidth-bound — it depends on how fast weights can be read from HBM.

Grouped-Query Attention (GQA)

Modern models use GQA where multiple query heads share fewer KV heads. This reduces KV cache size significantly. For example, Llama-3.1-70B has 64 attention heads but only 8 KV heads — an 8× reduction in KV cache compared to full multi-head attention.

3. Sizing Formulas

Weight Memory

For standard FP16/BF16 dense models:

Weight Memory (GB) = params_billion × 2

For models with mixed-precision or quantization (e.g. GPT-oss with MXFP4), an explicit override is used based on actual measured model size:

GPT-oss-120B example: Expert weights (MXFP4): ~114B params × 0.5 bytes/param = ~57 GB Non-expert weights (BF16): ~3B params × 2 bytes/param = ~6 GB Total ≈ 63 GB (verified: fits a single 80 GB H100)

KV Cache Size per Token

KV per token (bytes) = 2 × n_layers × n_kv_heads × head_dim × 2 ↑ ↑ K and V FP16 = 2 bytes

The head_dim is usually d_model / n_heads, but some models (like GPT-oss) specify it explicitly when the hidden size doesn't divide evenly.

Total Memory Footprint

Total Memory (GB) = Weight Memory + KV_per_token × context_window × concurrent_requests where: context_window = input_tokens + output_tokens
This is a worst-case estimate, assuming all concurrent requests use the full context window simultaneously. Real serving engines use paged attention and dynamic memory management for better utilization.

Minimum GPUs

Min GPUs = ⌈ Total Memory / GPU Memory ⌉

This indicates how many GPUs of a given type are needed purely for memory capacity. In practice, multi-GPU inference also requires tensor parallelism or pipeline parallelism communication overhead.

Prefill Time per Token

Prefill is compute-bound. The time to process each input token:

Prefill time/token (ms) = (2 × compute_params_billion) / GPU_FP16_TFLOPS

The factor of 2 accounts for the multiply-accumulate operations (1 multiply + 1 add per parameter). For MoE models, compute_params_billion = the active parameters per token, not total parameters.

TPOT — Time per Output Token

Decode is memory-bandwidth-bound. Each generated token requires reading the model weights from HBM:

TPOT (ms) = (2 × compute_params_billion) / memory_bandwidth_GBps × 1000

For MoE models, only the active expert weights are read per token, so we use active parameters.

TTFT — Time to First Token

TTFT (s) ≈ prefill_time_per_token + TPOT / 1000

This approximation assumes the first token is produced after processing the full prompt plus one decode step.

End-to-End Latency

E2E Latency (s) = (input_tokens × prefill_time + output_tokens × TPOT) / 1000

Total wall-clock time from sending the request to receiving the complete response.

Throughput

Throughput (tok/s) = output_tokens / E2E_latency

This is the single-request output token rate. Actual serving throughput with batching will be higher.

4. Mixture-of-Experts (MoE) Handling

Mixture-of-Experts models like GPT-oss use a routing mechanism where each token is processed by only a subset of "expert" sub-networks:

How MoE Affects Sizing

AspectDense ModelMoE Model
Memory All parameters in VRAM All parameters in VRAM (all experts must be loaded)
Compute per token All parameters Only active expert parameters (e.g., 5.1B out of 117B for GPT-oss-120B)
Bandwidth per token Read all weights from HBM Read only active expert weights from HBM
Net effect — Much faster inference despite larger total model size

GPT-oss MXFP4 Quantization

GPT-oss models use MXFP4 (Microscaling 4-bit floating point) quantization for the expert MLP weights, while keeping the rest in BF16. This is critical for accurate sizing:

GPT-oss-120B weight breakdown: 117B total parameters ├── ~114B in expert MLP weights → MXFP4 (0.5 bytes/param) = ~57 GB └── ~3B in shared/non-expert weights → BF16 (2 bytes/param) = ~6 GB Total weight memory ≈ 63 GB GPT-oss-20B weight breakdown: 21B total parameters ├── ~19B in expert MLP weights → MXFP4 (0.5 bytes/param) = ~9.5 GB └── ~2B in shared/non-expert weights → BF16 (2 bytes/param) = ~4 GB Total weight memory ≈ 14 GB
Verification: OpenAI states GPT-oss-120B is designed to run on a single H100 GPU (80 GB). With 63 GB weight memory, this leaves ~17 GB for KV cache and overhead — consistent with the claim.

5. Limitations & Caveats

These are theoretical first-order estimates. Real-world performance will differ due to the factors below.
  • Serving framework overhead: vLLM, TensorRT-LLM, SGLang, etc. each have different memory overhead for scheduling, paged attention metadata, CUDA graphs, and kernel launcher bookkeeping.
  • Continuous batching: The calculator models single-request latency. Real serving engines batch multiple requests, which improves throughput but may increase individual latency.
  • Tensor parallelism overhead: When using multiple GPUs, communication overhead (NVLink, InfiniBand) is not modeled. The "Min GPUs" column only reflects memory capacity.
  • Activation memory: During prefill, intermediate activations temporarily consume VRAM. This is not included in the static memory footprint.
  • System memory: CUDA context, driver overhead, and the serving framework itself consume a portion of VRAM (typically 1–3 GB).
  • Quantization accuracy: The 63 GB / 14 GB estimates for GPT-oss models are derived from the MXFP4 + BF16 breakdown. Actual checkpoint file sizes may differ slightly due to metadata, padding, and serialization format.
  • KV cache precision: We assume FP16 KV cache. Some frameworks support INT8 or FP8 KV cache, which halves this cost.
  • Speculative decoding, prefix caching, and other optimizations are not modeled.
